## Overview

Cric PR Backend API uses **JWT (JSON Web Tokens)** for stateless authentication, implemented with **Passport.js** and the `passport-jwt` strategy. This approach provides secure, scalable authentication without server-side session storage.

<Info>
  The authentication system supports multiple login methods: email/password, phone/password, Google OAuth, and Apple Sign-In.
</Info>

## Authentication Flow

<Steps>
  <Step title="User Registration or Login">
    Users register with credentials or login with existing accounts. The system validates credentials and creates/retrieves the user.
  </Step>
  
  <Step title="Token Generation">
    Upon successful authentication, the server generates two JWT tokens:
    - **Access Token**: Short-lived token for API requests (60 minutes)
    - **Refresh Token**: Long-lived token for obtaining new access tokens (30 days)
  </Step>
  
  <Step title="Making Authenticated Requests">
    Clients include the access token in the `Authorization` header using Bearer authentication:
    ```
    Authorization: Bearer <access_token>
    ```
  </Step>
  
  <Step title="Token Verification">
    The server extracts and verifies the JWT token from the request header. If valid, the user object is attached to the request.
  </Step>
  
  <Step title="Token Refresh">
    When the access token expires, clients use the refresh token to obtain a new access token without re-authenticating.
  </Step>
</Steps>

## JWT Strategy Implementation

The API uses Passport.js with a custom JWT strategy configured in `src/config/passport.js`:

```javascript src/config/passport.js
const { Strategy: JwtStrategy, ExtractJwt } = require('passport-jwt');
const config = require('./config');
const { tokenTypes } = require('./tokens');
const { User } = require('../models');

const jwtOptions = {
  secretOrKey: config.jwt.secret,
  jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
};

const jwtVerify = async (payload, done) => {
  try {
    // Verify token type is ACCESS
    if (payload.type !== tokenTypes.ACCESS) {
      throw new Error('Invalid token type');
    }
    
    // Find user by ID from token payload
    const user = await User.findById(payload.sub);
    if (!user) {
      return done(null, false);
    }
    
    done(null, user);
  } catch (error) {
    done(error, false);
  }
};

const jwtStrategy = new JwtStrategy(jwtOptions, jwtVerify);
```

<Note>
  The JWT strategy extracts tokens from the `Authorization` header as Bearer tokens and verifies them against the configured secret.
</Note>

## Token Types

The API uses four distinct token types defined in `src/config/tokens.js`:

```javascript src/config/tokens.js
const tokenTypes = {
  ACCESS: 'access',           // For API authentication
  REFRESH: 'refresh',         // For obtaining new access tokens
  RESET_PASSWORD: 'resetPassword',  // For password reset flow
  VERIFY_EMAIL: 'verifyEmail'      // For email verification
};
```

    **Storage**: Stored in database, blacklisted after use
  </Card>

    **Storage**: Stored in database with OTP hash
  </Card>

## Token Generation

Tokens are generated using the `jsonwebtoken` library in `src/services/token.service.js`:

```javascript src/services/token.service.js
const jwt = require('jsonwebtoken');
const moment = require('moment');

/**
 * Generate a JWT token
 */
const generateToken = (userId, expires, type, secret = config.jwt.secret) => {
  const payload = {
    sub: userId,              // Subject (user ID)
    iat: moment().unix(),     // Issued at time
    exp: expires.unix(),      // Expiration time
    type,                     // Token type
  };
  return jwt.sign(payload, secret);
};

/**
 * Generate access and refresh tokens
 */
const generateAuthTokens = async (user) => {
  // Generate access token (short-lived)
  const accessTokenExpires = moment().add(
    config.jwt.accessExpirationMinutes, 
    'minutes'
  );
  const accessToken = generateToken(
    user.id, 
    accessTokenExpires, 
    tokenTypes.ACCESS
  );

  // Generate refresh token (long-lived)
  const refreshTokenExpires = moment().add(
    config.jwt.refreshExpirationDays, 
    'days'
  );
  const refreshToken = generateToken(
    user.id, 
    refreshTokenExpires, 
    tokenTypes.REFRESH
  );
  
  // Store refresh token in database
  await saveToken(refreshToken, user.id, refreshTokenExpires, tokenTypes.REFRESH);

  return {
    access: {
      token: accessToken,
      expires: accessTokenExpires.toDate(),
    },
    refresh: {
      token: refreshToken,
      expires: refreshTokenExpires.toDate(),
    },
  };
};
```

<Warning>
  The JWT secret (`JWT_SECRET` environment variable) must be kept secure and never exposed in client-side code or public repositories.
</Warning>

## Authentication Middleware

Protected routes use the `auth()` middleware defined in `src/middlewares/auth.js`:

```javascript src/middlewares/auth.js
const passport = require('passport');
const httpStatus = require('http-status');
const ApiError = require('../utils/ApiError');
const { User } = require('../models');

const verifyCallback = (req, resolve, reject) => async (err, user, info) => {
  if (err || info || !user) {
    return reject(new ApiError(httpStatus.UNAUTHORIZED, 'Please authenticate'));
  }

  try {
    // Fetch full user with populated roles and permissions
    const fullUser = await User.findById(user._id).populate({
      path: 'roles',
      populate: { path: 'permissions' },
    });

    if (!fullUser) {
      return reject(new ApiError(httpStatus.UNAUTHORIZED, 'User not found'));
    }

    req.user = fullUser;  // Attach user to request
    resolve();
  } catch (error) {
    reject(new ApiError(httpStatus.INTERNAL_SERVER_ERROR, 'User population failed'));
  }
};

const auth = () => async (req, res, next) => {
  return new Promise((resolve, reject) => {
    passport.authenticate('jwt', { session: false }, verifyCallback(req, resolve, reject))(req, res, next);
  })
    .then(() => next())
    .catch((err) => next(err));
};
```

<Info>
  The middleware automatically populates the user's roles and permissions, making them available throughout the request lifecycle.
</Info>

```javascript
const login = catchAsync(async (req, res) => {
  const { email, phone, password, fcmToken } = req.body;

  if (!email && !phone) {
    return res.status(400).send({ message: 'Email or phone is required.' });
  }

  // Support both email and phone login
  const user = phone
    ? await authService.loginUserWithPhone(phone, password, fcmToken)
    : await authService.loginUserWithEmailAndPassword(email, password, fcmToken);

  const tokens = await tokenService.generateAuthTokens(user);
  
  // Update fcmToken if provided and check profile completeness
  let refreshedUser = user;
  if (fcmToken) {
    refreshedUser = await userService.updateUserById(user.id, { fcmToken });
  }
  
  const playerProfile = await playerService.getPlayerProfileByUserId(user._id);
  res.send({ user: refreshedUser, tokens, playerProfile });
});
```

<Note>
  The login endpoint automatically checks profile completeness and updates the player status to either `active` or `incomplete_profile`.
</Note>

## Password Reset Flow

The API implements a secure two-step password reset process using OTP verification.

<Steps>
  <Step title="Request Password Reset">
    Send email to receive OTP:
    
    ```bash
    curl -X POST http://localhost:3000/v1/auth/forgot-password \
      -H "Content-Type: application/json" \
      -d '{ "email": "john@example.com" }'
    ```
    
    **Response**: `{ "token": "hashed-token" }`
    
    Implementation: `src/controllers/auth.controller.js:94-102`
  </Step>
  
  <Step title="Verify OTP">
    Submit the OTP received via email:
    
    ```bash
    curl -X POST http://localhost:3000/v1/auth/verify-password \
      -H "Content-Type: application/json" \
      -d '{ "token": "hashed-token", "otp": "123456" }'
    ```
    
    **Response**: `{ "token": "reset-password-token" }`
  </Step>
  
  <Step title="Reset Password">
    Use the reset token to set a new password:
    
    ```bash
    curl -X POST "http://localhost:3000/v1/auth/reset-password?token=reset-token" \
      -H "Content-Type: application/json" \
      -d '{ "password": "newPassword123" }'
    ```
    
    Implementation: `src/controllers/auth.controller.js:104-107`
  </Step>
</Steps>

<Warning>
  In development mode, the OTP is always `123456` for testing purposes. In production, a random 6-digit code is generated.
</Warning>

## Security Best Practices

<CardGroup cols={2}>
  <Card title="Token Storage" icon="database">
    **Client-Side:**
    - Store access tokens in memory or sessionStorage (not localStorage for sensitive apps)
    - Store refresh tokens in httpOnly cookies when possible
    - Never expose tokens in URLs or logs
    
    **Server-Side:**
    - Refresh tokens are stored in MongoDB with user reference
    - Tokens can be blacklisted upon logout
    - Expired tokens are automatically invalidated
  </Card>
  
  <Card title="Password Requirements" icon="lock">
    Passwords must meet these criteria:
    - Minimum 8 characters
    - At least one letter (a-z or A-Z)
    - At least one number (0-9)
    
    Enforced by Joi validation in `src/validations/auth.validation.js`
  </Card>
  
  <Card title="Token Expiration" icon="clock">
    **Access Tokens**: 60 minutes
    - Short-lived to minimize security risk
    - Must be refreshed regularly
    
    **Refresh Tokens**: 30 days
    - Longer-lived for better UX
    - Stored in database for revocation
  </Card>
  
  <Card title="Rate Limiting" icon="shield">
    The API implements rate limiting on authentication endpoints:
    - Prevents brute force attacks
    - Configurable via `express-rate-limit`
    - Returns 429 status when limit exceeded
  </Card>
</CardGroup>

## Environment Configuration

Configure JWT settings via environment variables:

```bash .env
# JWT Secret (use a strong random string in production)
JWT_SECRET=your-super-secret-jwt-key-change-this-in-production

# Token expiration times
JWT_ACCESS_EXPIRATION_MINUTES=60
JWT_REFRESH_EXPIRATION_DAYS=30
JWT_RESET_PASSWORD_EXPIRATION_MINUTES=10
JWT_VERIFY_EMAIL_EXPIRATION_MINUTES=10
```

<Warning>
  **Critical Security Notes:**
  - Use a strong, randomly generated `JWT_SECRET` in production
  - Never commit `.env` files to version control
  - Rotate secrets periodically
  - Use different secrets for different environments
</Warning>

## Error Handling

The authentication system returns standard HTTP error codes:

| Status Code | Error | Description |
|-------------|-------|-------------|
| `400` | Bad Request | Invalid request body or validation failed |
| `401` | Unauthorized | Invalid credentials or expired token |
| `403` | Forbidden | Valid token but insufficient permissions |
| `404` | Not Found | User or resource not found |
| `500` | Internal Server Error | Server-side error during authentication |

```javascript Testing Suite
const BASE_URL = 'http://localhost:3000/v1';

// Test registration
async function testRegistration() {
```
