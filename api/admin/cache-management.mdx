## Notification Management

### Send Admin Notification

Send push notifications to all users or selected users.

<CodeGroup>
```bash cURL
curl -X POST \
  'https://api.cricpr.com/v1/admin/notifications/send' \
  -H 'Authorization: Bearer YOUR_ADMIN_TOKEN' \
  -H 'Content-Type: application/json' \
  -d '{
    "title": "New Feature Available",
    "body": "Check out our new statistics dashboard!",
    "action": "OPEN_STATS",
    "metadata": {
      "screen": "stats"
    }
  }'
```

```javascript JavaScript
const response = await fetch('https://api.cricpr.com/v1/admin/notifications/send', {
  method: 'POST',
  headers: {
    'Authorization': 'Bearer YOUR_ADMIN_TOKEN',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    title: 'New Feature Available',
    body: 'Check out our new statistics dashboard!',
    action: 'OPEN_STATS',
    metadata: {
      screen: 'stats'
    },
    userIds: null // Send to all users
  })
});
```
</CodeGroup>

### Request Body

<ParamField body="title" type="string" required>
  Notification title
</ParamField>

<ParamField body="body" type="string" required>
  Notification body text
</ParamField>

<ParamField body="metadata" type="object" optional>
  Additional data to send with notification (e.g., navigation info)
</ParamField>

<ParamField body="action" type="string" default="NONE">
  Action type for the notification (e.g., "OPEN_STATS", "VIEW_MATCH")
</ParamField>

<ParamField body="userIds" type="array" optional>
  Array of user IDs to send notification to. If null or empty, sends to all users with FCM tokens.
</ParamField>

### Response

<ResponseField name="success" type="boolean">
  Whether notification was sent successfully
</ResponseField>

<ResponseField name="message" type="string">
  Success or error message
</ResponseField>

<ResponseField name="sentCount" type="number">
  Number of users notification was sent to
</ResponseField>

### Example Response

```json
{
  "success": true,
  "message": "Notification sent to 1543 user(s)",
  "sentCount": 1543
}
```

### Notes

- Only one admin notification record is saved (not per-user)
- Notifications are sent in batches of 500 for performance
- Users without valid FCM tokens are skipped
- Deleted users are excluded automatically

## Cache Management

While there are no explicit cache clearing endpoints, the system manages caching automatically:

### Automatic Cache Updates

Caches are updated automatically when:

1. **Match Statistics Update**: Stats API updates trigger cache refresh
2. **Match Completion**: Match summaries are recalculated
3. **Data Modifications**: Updates to players, teams, or matches invalidate related caches

### Match Summary Caching

Match summaries are cached for performance. From the codebase:

```javascript
// Get match summary with cache skip option
await matchService.getMatchSummary(matchId, { skipCache: true });
```

The admin endpoints automatically skip cache when retrieving fresh data.

### Statistics Caching

Player and team statistics are calculated on-demand but cached:

1. **Player Match Stats**: Cached per match-player combination
2. **Team Totals**: Aggregated and cached per team
3. **Platform Stats**: Cached globally (sixes, fours, extras)

### Cache Invalidation

Caches are invalidated when:

- Statistics are updated via Stats API
- Matches are completed or modified
- Players or teams are updated
- Data is deleted via admin endpoints

### Best Practices

1. **Update Statistics**: Use Stats API to update match stats when needed
2. **Force Refresh**: Admin detail endpoints fetch fresh data
3. **Monitor Performance**: Check dashboard metrics regularly
4. **Batch Operations**: Update multiple items together for efficiency

## Data Management

### Soft Deletes

The platform uses soft deletes for most entities:

```javascript
{
  status: 'deleted'
}
```

Deleted entities are:
- Excluded from list queries
- Excluded from statistics calculations
- Still referenced in historical data
- Not physically removed from database

### Hard Deletes

Admin delete endpoints perform hard deletes:
- **Delete Team**: Removes team and all team instances
- **Delete Match**: Removes match, innings, balls, and statistics
- **Delete Tournament**: Removes tournament and associations

<Warning>
  Hard deletes cannot be undone. Always verify data before deletion.
</Warning>

### Data Dependencies

Before deleting, check dependencies:

1. **Teams**: Check for active matches and tournaments
2. **Matches**: Check for tournament associations
3. **Tournaments**: Check for ongoing matches
4. **Players**: Cannot be deleted directly (use soft delete)

### Backup Recommendations

1. Export data before mass deletions
2. Keep audit logs of admin actions
3. Regular database backups
4. Test delete operations in staging first

## Notification System

### FCM Token Management

The notification system uses Firebase Cloud Messaging (FCM):

- Users register FCM tokens on login
- Tokens stored in user profiles
- Invalid/empty tokens automatically filtered
- Tokens refreshed on app updates

### Notification Batching

From `admin.service.js:src/services/admin.service.js:2563-2572`:

```javascript
await notificationService.sendNotificationsToUsers(
  usersWithTokens,
  title,
  body,
  metadata,
  action,
  500, // batchSize
  false, // saveToDB
  false // isAdmin
);
```

Notifications are sent in batches of 500 for optimal performance.

### Notification Actions

Supported action types:
- `NONE`: No specific action
- `OPEN_STATS`: Navigate to statistics
- `VIEW_MATCH`: Open match details
- `VIEW_TOURNAMENT`: Open tournament
- Custom actions defined by mobile app

### Metadata Format

Metadata can include navigation info:

```json
{
  "metadata": {
    "screen": "matchDetails",
    "matchId": "60f7b3b3b3f3b3b3b3f3b3b3",
    "tab": "scorecard"
  }
}
```


```json
{
```

```json
  "error": "Database error",
  "code": 500,
  "meta": {

  }
}
```

Some endpoints use different error formats:

**Standard Format**
```json
{

}
```

```json

  "error": "Error message",
```
